# QA Document Audit

This section details the QA Document Audit.

## Version
1.0.0

## Engineering Intent
Verifies a QA document's test types cohere as one strategy — Test Strategy's priority mapping is actually reflected in Unit/Integration/E2E/Load/Scalability/Security Testing, not a plan that the rest of the document ignores. Section-level quality per test type is owned by `audit/semantic/section/qa/`; this file owns whether the whole test plan is internally consistent.

## Audit Objectives
- Test Strategy's priority levels and project profile mapping are reflected in which test types are actually detailed (a high-priority type shouldn't be thin, a deprioritized one shouldn't be over-built)
- Integration Testing's component mapping is consistent with Architecture(05) — no tested boundary that Architecture doesn't define, no defined boundary left untested
- E2E Testing's journeys map to Design(06)'s documented workflows, not invented independently
- Terminology consistent across all test type sections — the same test type or coverage concept isn't named differently in different sections

## Script Evidence Grounding

When available, the following script outputs provide ground-truth context for this audit. The LLM evaluator should use these as factual anchors rather than relying solely on what the document claims.

| Script | Evidence field | How it grounds the audit |
|--------|---------------|------------------------|
| `unit-test-coverage` | `metrics.coverage_percent`, `evidence[]` | Validates or contradicts the document's claimed test coverage. If the doc says "85% coverage" but the script reports 62%, that's a grounding conflict to flag. The `evidence` array lists untested files — useful for assessing whether the document's coverage strategy is actually implemented. |

When script evidence is available, the evaluator should:
1. Compare script-reported metrics against document claims
2. Flag contradictions where script ground-truth differs from doc assertions
3. Use script `evidence` arrays as concrete examples when scoring criteria about implementation completeness

## Expected Quality
- A test type Test Strategy marks high-priority has correspondingly detailed content in its own section
- Integration Testing's boundaries trace to specific Architecture components, not vague "the API layer"
- E2E journeys are traceable to specific Design workflows

## Red Flags
- Test Strategy claims a risk-based justification but the detailed sections don't reflect that prioritization at all
- Integration Testing tests boundaries that don't exist in Architecture, or ignores boundaries Architecture defines
- E2E Testing invents user journeys with no corresponding Design workflow
- Same coverage concept (e.g. "critical path") means different things across test type sections

## Edge Cases
- QA document for a system with no formal Design workflows yet (early-stage project) — E2E section should state that gap rather than fabricate journeys
- Deliberately deprioritized test type (e.g. Load Testing skipped for a low-traffic internal tool) — acceptable if Test Strategy states the rationale, not silently thin

## Scoring Criteria

| ID | Weight | Score | Description |
|---|---|---|---|
| C1 | mandatory | 0 or 40 | Test Strategy's priorities are reflected in the depth of each test type section |
| C2 | mandatory | 0 or 30 | Integration Testing boundaries consistent with Architecture(05) |
| C3 | recommended | 0 or 30 | E2E journeys traceable to Design(06) workflows |

Score = sum of passed criterion scores, capped at 100.
Mandatory criterion failure = ERROR. Recommended = WARNING.
