# Workflows

## Purpose

Workflows are phased project plans generated by knowledge systems. They define the sequence of operations (generate, audit, fix, verify) that transform a repository from its current state to a desired state.

## What Workflows Are

A workflow is a plan with ordered phases, each targeting specific domains and capabilities. A system's `init` script returns the full set of workflows it supports (§8.4's `InitPlan` shape); `store_system_plan` splits that into the `workflow_use_cases`/`workflow_phases`/`workflow_phase_dependencies` tables — real rows, not a JSON blob column.

## Workflow Structure

```
Workflow
  ├── Phase 1: Generate (compile documentation)
  ├── Phase 2: Audit (validate all domains)
  ├── Phase 3: Fix (address findings)
  ├── Phase 4: Verify (re-audit, check gates)
  └── Phase 5: Report (generate summary)
```

## Phase Types

| Phase Type | What It Does | System Script |
|---|---|---|
| `generate` | Compile documentation | `compile()` (built-in) |
| `audit` | Run validation checks | `validate` |
| `fix` | Apply fixes to findings | `validate` (re-audit) |
| `verify` | Re-run checks, compare thresholds | `validate` |
| `report` | Generate summary reports | `report` |

## Phase Dependencies

Phases can declare dependencies on other phases:

```json
[
  { "id": "audit-doc", "kind": "script", "depends_on": [] },
  { "id": "fix-doc", "kind": "semantic", "depends_on": ["audit-doc"] },
  { "id": "verify", "kind": "script", "depends_on": ["fix-doc"] }
]
```
Once stored (`store_system_plan`), each `depends_on` entry becomes a real
row in `workflow_phase_dependencies` — not kept as a JSON array server-side.

## Prerequisites

Before a phase executes, samgraha checks its prerequisites:

- All `depends_on` phases must be completed
- TTL expiry rules must not be violated
- Head commit must match (if configured)

If prerequisites are not met, the phase is blocked and returns a blocked-precondition response.

## Init Plans

A system's workflows are **not** declared statically in `system.yaml` —
`system.yaml` only ever holds `name`/`description`/`extends`/`drops`/
`abstract` (see [Knowledge Systems](systems.md)). Workflows come from
running the system's `init` **script** (`scripts/init.py`), which returns
the plan as JSON, and that JSON is what `store_system_plan` accepts:

```json
{
  "system": "rust_dev",
  "use_cases": [
    {
      "id": "new-repo",
      "label": "New repo, no code, no docs",
      "phases": [
        {
          "id": "vision-plan-input",
          "kind": "semantic",
          "description": "Determine what Vision needs, given repo state",
          "depends_on": []
        },
        {
          "id": "vision-plan-render",
          "kind": "script",
          "script": "scripts/plan_generation.py",
          "depends_on": ["vision-plan-input"],
          "expiry": null
        }
      ]
    }
  ]
}
```
`kind: "semantic"` phases have no `script` field — there's nothing to run;
the LLM does that work via MCP. `kind: "script"` phases name the script to
dispatch. `depends_on` lists other phase ids, resolved into real
`workflow_phase_dependencies` edges once stored — not kept as a JSON array.
